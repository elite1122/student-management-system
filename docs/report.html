1. Introduction

In modern educational institutions, managing student records manually is time-consuming, error-prone, and inefficient. As the number of students grows, the need for a digital solution becomes essential. A Student Management System provides a centralized platform to store, manage, and access student information in an organized and secure manner.

This project presents a full-stack Student Management System built using modern web development technologies. The application provides a clean, professional user interface to perform all CRUD (Create, Read, Update, Delete) operations on student records. It features a React-based frontend with Tailwind CSS styling and smooth animations, an Express.js RESTful API backend, and a MongoDB Atlas cloud database for persistent storage.

Beyond the core functionality, this project also demonstrates industry-standard practices including Docker containerization for consistent deployment environments and a CI/CD pipeline using GitHub Actions that automatically builds and deploys the application to Vercel on every code push. This report documents the complete technical implementation, architecture decisions, tools used, and deployment strategy.

3. Methodology

This project was developed following the Agile methodology with an incremental approach. The development was divided into distinct phases, each building upon the previous one. Below is the step-by-step methodology followed throughout the project:

Phase 1: Planning & Requirement Analysis (Day 1)

  • Identified the core requirements: CRUD operations for student records
  • Selected the tech stack: React, Node.js, Express, MongoDB, Tailwind CSS
  • Defined the database schema (student fields, validation rules)
  • Planned the project folder structure (separate frontend and backend directories)
  • Set up the development environment and initialized Git repository

Phase 2: Backend Development (Day 2-3)

  • Created the Express.js server with CORS and JSON middleware
  • Designed the Mongoose Student schema with validations
  • Implemented RESTful API endpoints (GET, POST, PUT, DELETE)
  • Added search, filter, and pagination functionality to the GET endpoint
  • Created a global error handler for validation errors, duplicate keys, and invalid IDs
  • Connected to MongoDB Atlas cloud database
  • Tested all API endpoints using Postman/browser

Phase 3: Frontend Development (Day 3-5)

  • Set up React project with Vite build tool
  • Configured Tailwind CSS for styling and custom design tokens
  • Built UI components: Navbar, StatsCard, StudentTable, StudentModal, DeleteConfirmModal
  • Integrated Framer Motion for page animations and transitions
  • Created Axios API service for backend communication
  • Implemented state management using React hooks (useState, useEffect, useCallback)
  • Added search bar, status filter, pagination, and toast notifications

Phase 4: Docker & Containerization (Day 5-6)

  • Created multi-stage Dockerfile for the backend (Node.js + Alpine)
  • Created multi-stage Dockerfile for the frontend (Node build + Nginx serve)
  • Configured Nginx for API proxying and SPA routing
  • Created docker-compose.yml to orchestrate both services
  • Tested the full application running in Docker containers

Phase 5: CI/CD & Deployment (Day 6-7)

  • Created Vercel projects for both frontend and backend
  • Configured Vercel serverless functions for the Express backend
  • Set up GitHub Actions workflow for automated deployment
  • Configured GitHub Secrets for sensitive credentials
  • Tested the full CI/CD pipeline — push triggers build and deploy
  • Verified live deployment on Vercel

Phase 6: Documentation & Testing (Day 7)

  • Wrote project README.md for the GitHub repository
  • Prepared this comprehensive project report
  • Performed final end-to-end testing of all features

12. Docker & Containerization

Docker packages the application into isolated containers so it runs identically on any machine, regardless of the operating system or installed software. This project uses multi-stage Docker builds for optimized image sizes and Docker Compose for orchestrating multiple services.

12.1 Backend Dockerfile

Uses a multi-stage build: installs production dependencies in a builder stage, then copies them to a lean Alpine image. Runs as a non-root user for security.

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

FROM node:20-alpine
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY . .
USER appuser
EXPOSE 5000
HEALTHCHECK CMD wget -qO- http://localhost:5000/health || exit 1
CMD ["node", "server.js"]

12.2 Frontend Dockerfile

Builds the React application with Vite, then serves the static files using Nginx web server.

FROM node:20-alpine AS builder
ARG VITE_API_URL=/api
ENV VITE_API_URL=$VITE_API_URL
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

12.3 Docker Compose

Orchestrates both containers on a shared network. The frontend Nginx reverse-proxies /api requests to the backend container.

services:
  backend:
    build: ./backend
    container_name: sms-backend
    ports: ["5000:5000"]
    environment:
      - MONGO_URI=${MONGO_URI}

  frontend:
    build: ./frontend
    container_name: sms-frontend
    ports: ["80:80"]
    depends_on:
      backend:
        condition: service_healthy

13.2 Workflow Configuration

The CI/CD pipeline is defined in .github/workflows/ci-cd.yml:

name: CI/CD — Build & Deploy to Vercel

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  deploy-backend:
    name: Deploy Backend
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm install -g vercel@latest
      - name: Deploy Backend to Vercel
        run: vercel deploy --prod --yes --token=${{ secrets.VERCEL_TOKEN }}
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_BACKEND_PROJECT_ID }}

  deploy-frontend:
    name: Deploy Frontend
    runs-on: ubuntu-latest
    needs: deploy-backend
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
        working-directory: ./frontend
      - run: npm run build
        working-directory: ./frontend
        env:
          VITE_API_URL: ${{ secrets.VITE_API_URL }}
      - run: npm install -g vercel@latest
      - name: Deploy Frontend to Vercel
        run: vercel deploy --prod --yes --token=${{ secrets.VERCEL_TOKEN }}
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_FRONTEND_PROJECT_ID }}

14. Deployment (Vercel)

Backend — Vercel Serverless Functions

The Express.js backend runs as a Vercel Serverless Function. The vercel.json routes all requests to server.js. MongoDB connections are cached between function invocations for performance optimization.

Frontend — Vercel Static Hosting

The React frontend is built by Vite into static files (HTML, CSS, JS) and deployed to Vercel's global CDN. The vercel.json rewrite ensures all routes serve index.html.
